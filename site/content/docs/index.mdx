SlotPilot is an AI-assisted Solana transaction control tower.

Most Solana apps stop at `sendTransaction`. SlotPilot treats submission as an operational lifecycle:

- observe network state
- submit through Jito bundles
- verify commitment progression
- classify failures
- retry from observed context
- prove every attempt with lifecycle logs

| Capability | What it does |
| --- | --- |
| Yellowstone live stream | Reads live slot and transaction updates |
| Jito bundle submission | Submits signed transactions through the block engine |
| Dynamic tip oracle | Selects tip account and tip size per attempt |
| AI retry agent | Returns structured retry, hold, or abort decisions |
| Lifecycle logs | Stores append-only proof for every attempt |

## 1. Problem Statement

Sending a Solana transaction is only one part of the process.

Before a transaction lands, it passes through leader scheduling, TPU ingestion, block production, shred propagation, and commitment progression. Production systems need to understand these stages instead of only relying on RPC polling.

The key challenge is simple:

> A Jito bundle id does not prove the transaction landed.

SlotPilot tracks the transaction through `processed`, `confirmed`, and `finalized` states before treating the attempt as proven.

## 2. Architecture Overview

SlotPilot is a modular monolith with an external AI decision agent.

The transaction engine observes, builds, signs, submits, and tracks. The AI agent only receives observed context and returns an operational decision. It does not hold private keys and does not sign transactions.

### Blockhash Commitment Rule

Finalized blockhashes are older and reduce the usable lifetime of a transaction. For time-sensitive submissions, the system should use `processed` or `confirmed` commitment.

### AI Agent Response

```json
{
  "decision": "retry",
  "reason": "The transaction failed because the blockhash expired before landing.",
  "refreshBlockhash": true,
  "tipMultiplier": 1.2,
  "delaySlots": 0
}
```

## 9. README Questions

### What does the delta between `processed_at` and `confirmed_at` tell you about network health at the time of submission?

The delta between `processed_at` and `confirmed_at` shows how quickly the network moved from local block processing to broader vote confirmation. A small delta suggests healthy propagation and voting. A large delta can indicate congestion, slow propagation, fork uncertainty, skipped slots, or validator voting delays.

### Why should you never use finalized commitment when fetching a blockhash for a time-sensitive transaction?

Finalized commitment gives an older blockhash because it waits for the block to become rooted. For time-sensitive transactions, an older blockhash reduces the remaining lifetime before expiry. Processed or confirmed commitment gives a fresher blockhash and more time for the transaction to land.

### What happens to your bundle if the Jito leader skips their slot?

If the Jito leader skips their slot, the bundle may not land in that slot. The system must detect that the transaction did not progress, refresh the blockhash if expiry risk increases, recalculate the tip, and resubmit during another valid leader window.
